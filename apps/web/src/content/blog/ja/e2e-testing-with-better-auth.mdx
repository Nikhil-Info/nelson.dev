## TL;DR

有効なセッショントークンを生成し、クッキーとして保存する必要があります。これにより、E2Eテストが手動ログインなしで認証済みルートにアクセスできます。この方法は、資格情報ベースとOAuth認証の両方で機能します。

> GitHubリポジトリ: [nelsonlaidev/e2e-testing-with-better-auth](https://github.com/nelsonlaidev/e2e-testing-with-better-auth)

## はじめに

このガイドでは、E2Eテストフレームワークとして[Playwright](https://playwright.dev/)を使用しますが、この概念は[Cypress](https://www.cypress.io/)などの他のフレームワークにも適用できます。

簡単にするために、資格情報ベースの認証の例を使用します。ただし、同じ原則がOAuthプロバイダーにも適用されます。

また、デモンストレーション目的でSQLiteデータベースを使用します。データベースの操作（例：テーブル名、カラムタイプ）を設定に合わせて調整してください。

## セッショントークンの生成

認証済みセッションをシミュレートするには、`BETTER_AUTH_SECRET`を使用して署名済みセッショントークンを生成する必要があります。このトークンはセッションクッキーで使用されます。

```ts
import crypto from 'node:crypto'

export const TEST_USER = {
  name: 'テストユーザー',
  email: 'test@example.com',
  sessionToken: '00000000000000000000000000000000',
  accountId: '000'
}

const signature = crypto
  .createHmac('sha256', process.env.BETTER_AUTH_SECRET!)
  .update(TEST_USER.sessionToken)
  .digest('base64')
const signedValue = `${TEST_USER.sessionToken}.${signature}`
```

## テストデータの挿入

テスト中に複数のユーザーを作成するのを避けるために、テストユーザーを静的に保ちたいと考えています。まだ存在しない場合は、テストユーザー、アカウント、セッションをデータベースに挿入します。簡単にするために、すべての主キーに対して一意のIDとして`0`を使用します。

```ts
import { db } from '@/lib/db'

const TEST_UNIQUE_ID = '0'

const now = new Date().getTime()
const expiresAt = new Date(Date.now() + 7 * 24 * 60 * 60 * 1000).getTime() // 7日間

const transaction = db.transaction(() => {
  db.prepare(
    `
      INSERT OR IGNORE INTO user (
        id, name, email, emailVerified, createdAt, updatedAt
      ) VALUES (?, ?, ?, ?, ?, ?)
    `
  ).run(TEST_UNIQUE_ID, TEST_USER.name, TEST_USER.email, 0, now, now)

  db.prepare(
    `
      INSERT OR IGNORE INTO account (
        id, accountId, providerId, userId, password, createdAt, updatedAt
      ) VALUES (?, ?, ?, ?, ?, ?, ?)
    `
  ).run(TEST_UNIQUE_ID, TEST_USER.accountId, 'credential', TEST_UNIQUE_ID, 'password_hash', now, now)

  db.prepare(
    `
      INSERT INTO session (
        id, token, userId, expiresAt, createdAt, updatedAt
      ) VALUES (?, ?, ?, ?, ?, ?)
      ON CONFLICT(token) DO UPDATE SET
      expiresAt = excluded.expiresAt,
      updatedAt = excluded.updatedAt
    `
  ).run(TEST_UNIQUE_ID, TEST_USER.sessionToken, TEST_UNIQUE_ID, expiresAt, now, now)
})

transaction()
```

## セッションの保存

署名済みトークンをテストフレームワークで使用するためにJSONファイルにクッキーとして保存します。クッキー値を`encodeURIComponent{:.entity.name.function}`でエンコードすることを忘れないでください。

```ts
import fs from 'node:fs/promises'

const cookieObject = {
  name: 'better-auth.session_token',
  value: encodeURIComponent(signedValue), // [!code highlight]
  domain: 'localhost',
  path: '/',
  httpOnly: true,
  secure: false,
  sameSite: 'Lax',
  expires: Math.round(expiresAt / 1000)
}

await fs.writeFile('.auth/auth.json', JSON.stringify({ cookies: [cookieObject], origins: [] }, null, 2))
```

## テストでのセッションの使用

ログインせずに保護されたルートにアクセスするために、保存されたセッションをE2Eテストで読み込みます。以下の例ではPlaywrightを使用していますが、これはクッキーインジェクションをサポートする任意のテストライブラリ（例：Cypress）に適応できます。

```ts title='playwright.config.ts'
export default defineConfig({
  // ...
  projects: [
    { name: 'setup', testMatch: /global\.setup\.ts/, teardown: 'teardown' }, // [!code highlight]

    { name: 'teardown', testMatch: /global\.teardown\.ts/ }, // [!code highlight]

    {
      name: 'chromium',
      use: {
        ...devices['Desktop Chrome'],
        storageState: '.auth/auth.json' // [!code highlight]
      },
      dependencies: ['setup'] // [!code highlight]
    }
  ]
})
```
